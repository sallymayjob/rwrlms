# Launch Guide

## Goal
Deploy a complete Slack-native LMS without external infrastructure.

## Prerequisites

- Google Workspace account with Apps Script + Sheets access
- Slack workspace with app creation permissions
- CSV content generated by LMS Composer

## Launch Checklist

1. Create `RWR_LMS_DATABASE` sheet and tabs.
2. Import CSV templates from `/database`.
3. Configure Apps Script project from `/apps-script` files.
4. Set Script Properties for secrets and IDs.
5. Deploy Web App endpoint.
6. Create Slack app from manifest and install to workspace.
7. Set slash command endpoints.
8. Validate command flow with pilot users.

## Trigger Types Used + Provisioning Steps

Provision from **Apps Script → Triggers** unless noted otherwise.

| Trigger type | Function | How to provision | Expected payload schema |
| --- | --- | --- | --- |
| Time-driven | `sendDailyLesson` | Add Trigger → Function: `sendDailyLesson` → Event source: Time-driven → daily schedule. | No event payload required (`{}`); function pulls active learners/course state from Sheets. |
| Time-driven | `weeklyLeaderboard` | Add Trigger → Function: `weeklyLeaderboard` → Event source: Time-driven → weekly schedule. | No event payload required (`{}`); function reads learner progress and posts summary. |
| Time-driven | `progressReport` | Add Trigger → Function: `progressReport` → Event source: Time-driven → weekly schedule. | No event payload required (`{}`); function computes aggregate progress data. |
| Webhook | `doPost(e)` | Deploy Web App and configure Slack command/events/interactive URLs to deployment endpoint. | `e.postData.contents` parsed as either JSON Events payload (`{type,event,...}`) or form-encoded slash payload (`{command,text,user_id,channel_id,trigger_id,response_url,...}`) or interactive wrapper (`payload=<JSON>`). |
| onEdit / onChange installable | Not currently provisioned | None required now. If introduced, add spreadsheet installable triggers (not simple triggers) for stable auth + logging behavior. | Future shapes: `onEdit(e)` includes edited `range`/`value`/`oldValue`; `onChange(e)` includes `changeType` + `source`. |

## Operational Guidance

- **Retries**
  - Keep `doPost(e)` responses fast so Slack retries are minimized.
  - Monitor Apps Script Executions for failed time-driven jobs and rerun after remediation.
- **Idempotency keys**
  - Webhook: persist `event_id` (or `trigger_id`) before side effects; fallback to deterministic hash of request body + timestamp bucket.
  - Scheduler: use deterministic run IDs such as `sendDailyLesson:2026-03-06` or `weeklyLeaderboard:2026-W10`.
- **Duplicate-trigger protection**
  - Before writes or Slack sends, check whether key already exists in `Logs` (or dedicated idempotency table).
  - Skip duplicate work and record a duplicate-suppressed log event for audit trails.

## Verification Checklist (Trigger Button → Workflow → Sheets IO → Completion)

1. In **Apps Script → Triggers**, verify each trigger row exists once with correct schedule/source.
2. Trigger workflow start:
   - Run one scheduled function manually (`sendDailyLesson`), and
   - invoke one Slack command (`/learn` or `/progress`) to hit `doPost`.
3. Confirm **Sheets duplex IO**:
   - Reads happen from source tabs (e.g., `Learners`, `Courses`, `Lessons`).
   - Writes/logs appear in destination tabs (e.g., `Submissions`, `Logs`).
4. Confirm completion logs include function name, timestamp, and success/error outcome.
5. Re-run same payload/window and verify duplicate protection suppresses second processing.

## Pilot Validation Script

- `/onboard`
- `/courses`
- `/enroll C001`
- `/learn`
- `/submit M03-W02-L04 complete`
- `/progress`

Expected result: profile created, enrollment succeeds, lesson delivered, submission tracked, progress increased.
